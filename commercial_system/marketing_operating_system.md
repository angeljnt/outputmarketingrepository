# Marketing Operating System

## Objective

Define how Output Marketing runs product and marketing on an ongoing basis.

## Core assets

- `commercial_canon.md`
- `client_taxonomy.md`
- `clients_master.csv`
- `services_master.md`
- `packages_master.md`
- `proof_inventory.md`
- `selling_proof_library.md`
- website copy in `outputmarketing/src/pages/`
- outreach playbooks and targeting docs

## Channel roles

| Channel | Job |
|---|---|
| Website | package clarity, conversion, proof, trust |
| Work / case studies | proof and risk reduction |
| Pricing | package explanation and add-on logic |
| Contact | conversion and qualification |
| Outreach | first-touch relevance and ICP expansion |
| Proposals | tailored conversion layer after interest exists |
| CRM | operating memory and pipeline discipline |

## Weekly cadence

- review new leads against `client_taxonomy.md`
- assign package fit and proof angle in CRM
- update outreach based on live objections and live responses
- check whether website and outreach language are still aligned

## Monthly cadence

- review which package is being sold most often
- review which objections keep appearing
- review proof gaps
- update `selling_proof_library.md` with any new usable snippets
- prune low-fit ICP experiments

## Quarterly cadence

- review whether the core public offer still reflects real demand
- review whether any custom work should become more visible
- review whether any visible offer should be downgraded or hidden
- update the website if package logic changes

## Minimum CRM schema

Every active lead should carry these fields:

- company name
- segment
- buyer type
- stage fit
- offer fit
- primary package fit
- secondary package fit
- main pain
- proof angle
- outreach status
- next step
- notes

## Decision rules

### When to widen the offer

Only widen the public offer when:

- demand repeats
- proof exists
- delivery is repeatable
- the new offer does not dilute the core B2B SaaS story

### When to keep something custom

Keep it custom when:

- the work is real but inconsistent
- proof is partial
- the buyer type is secondary
- the service would clutter the public pricing structure

### When to update the website

Update the website whenever:

- package names change
- package scope changes
- add-ons change
- the homepage promise changes
- white-label visibility changes
